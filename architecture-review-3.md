## ⚠️ 本轮微调项

### 1. 低风险：updatedAt 精确相同时需要 tie-breaker

**问题**: 两个设备在同一毫秒更新同一字段时，无法判断谁更新。

**建议**: 在数据结构中追加 `deviceId` 作为次排序键：
```json
{ "value": "新标题", "updatedAt": "...", "deviceId": "mac-xxx" }
```
相同 `updatedAt` 时按 `deviceId` 字典序取较大者。

### 2. 低风险：推送应只推送给非来源设备

**问题**: Mac 端创建了待办，后端通过 WebSocket 又推送给 Mac 端自己，造成冗余处理。

**建议**: 后端推送时跳过来源设备（根据 WebSocket 连接对应的 token 判断）。

### 3. 低风险：缺少权限说明

**问题**: macOS 和鸿蒙端需要特定系统权限，架构描述中未提及。

**建议**: 添加各端权限清单：

**macOS（Info.plist）**:
- `NSCalendarsUsageDescription` - 读取日历
- `NSRemindersUsageDescription` - 读取提醒事项

**鸿蒙（module.json5）**:
- `ohos.permission.INTERNET` - 网络请求

### 4. 信息项：添加健康检查接口

建议增加 `GET /api/health` 返回 `{ status: "ok" }`，方便联调时确认服务是否运行。

### 5. 信息项：统一错误码格式

建议补充 HTTP 状态码的对应：
- `200` - 成功
- `400` - 请求参数错误
- `401` - Token 无效/过期
- `404` - 资源不存在
- `500` - 服务端错误
